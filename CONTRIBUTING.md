#  Методологии и codestyle

Здесь коротко описаны методологии разработки, используемые в проекте, начиная с main_1.2.0, а также другие нюансы именования и codestyle.

- GitHub Flow для управления ветками
- Conventional Commits для написания коммитов
- Semantic Versioning для версирования
- Флаг no-deploy

## GitHub Flow

- Основная ветка main всегда стабильна и готова к деплою.
- Для каждой задачи создаётся отдельная ветка с описательным названием (например, refactor-authentication). 
- В коммитах фиксируются изменения последовательно. В идеале каждый коммит — отдельная единица изменения, что упрощает откат при необходимости.
- После завершения работы открывается Pull Request для ревью и слияния.

## Conventional Commits

Структура сообщения: type[optional scope]: <description> [optional body] [optional footer(s)].  

Типы коммитов:
- feat: добавление новой функциональности
- fix: исправление ошибок
- docs: изменения в документации
- style: изменения стиля кода (форматирование без изменения логики)
- refactor: изменение кода без добавления функций или исправления ошибок
- perf: оптимизация производительности
- test: добавление или изменение тестов
- build: изменения в системе сборки
- chore: вспомогательные изменения (конфиги, скрипты)
- ci: изменения конфигурации CI/CD
- revert: отмена предыдущего коммита  

Используйте повелительное наклонение, избегайте «я», будьте краткими, но информативными.

Пример:
```
feat[backend]: создать базу данных

Реализован новый способ хранения данных.
Используем MySQL.

BREAKING CHANGE: Удалены старые обработчики json-файлов.
```

## Semantic Versioning

Формат версии: MAJOR.MINOR.PATCH

- Увеличивайте MAJOR, если внесены изменения, нарушающие обратную совместимость.
- Увеличивайте MINOR, если добавлены новые функции с сохранением совместимости.
- Увеличивайте PATCH, если исправлены ошибки без изменения функциональности.

Версирование применяется к PR в main.

## Флаг no-deploy

Если Pull Request не затрагивает файлов, необходимых для продакшена, то в заголовке PR используется флаг [no-deploy]. Например:

```
main_x.y.z создать XXX.md [no-deploy]
```

По методологии GitHub Flow прямой пуш в main не предусмотрен, но если по каким-либо причинам такое необходимо, то [no-deploy] прописывается в заголовке коммита:

```
main_x.y.z docs[root]: создать XXX.md [no-deploy]
```
